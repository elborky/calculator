# M2 UI Spec ‚Äî Keypad Screen (build-ready brief for the 3 mockup variants)

> **What this file is.** The concrete, build-ready UI specification for the M2 keypad screen ‚Äî the
> single screen that realizes the 8 flows (`02-flows.md`) and satisfies the 32 UR rules
> (`03-rules.md`), rendered in the owner-locked glassmorphism look (`08-design-system.md`). It is a
> **SPEC, not code** ‚Äî every visual is tied to a design-system section (`¬ßN`) or a UI rule (`UR-NNN`).
> The 3 mockup variants (Step 5.5) are the *code*; this file is the brief they are generated from.
>
> **Validated UX interpretation (owner-confirmed, inline verbatim ‚Äî orchestrator carried this in).**
> Owner: *"simple aja, tapi secara design ga bosenin"* ‚Üí glassmorphism, frosted glass over gradient.
> Interpreted (and owner-validated) as: a **single-panel, center-stage** calculator ‚Äî one glass body
> floating centered on the gradient; **display at top** (large primary number + a smaller pending-
> expression line above it); **keypad below** as a 4-column grid; buttons are **individual glass
> panes**; operators **coral** (`--operator-accent`), equals **indigo** (`--accent`). This M2 screen has
> **NO history panel** (M3) and **NO theme toggle** (M4) ‚Äî but the layout **reserves room** for both
> (history slot per design ¬ß6; toggle is M4's). Owner also locked a mockup-vibe steer (taste decision):
> **"DORONG SAMPAI MENTOK" / push to the limit** ‚Äî so the 3 variants must span a *wide* range (¬ß9).
>
> **Scope caveat carried from the hero mockup (`09-hero-mockup.html` header, owner-locked 2026-06-14).**
> The hero render showed **`¬±` and `%`** keys ‚Äî those are **ILLUSTRATIVE ONLY and OUT OF SCOPE**
> (`04-scope.md`: basic 4-function only). **This M2 screen does NOT build `¬±` or `%`.** The two grid
> cells the hero gave them are reclaimed for **AC** and **CE** (M2 needs both clears ‚Äî `02-flows.md`
> ¬ß5; the hero only showed AC). This is the one deliberate divergence from the hero layout, and it is
> required, not optional.

## 1. Overall composition ‚Äî single glass body, center-stage

The whole screen is **one glass calculator body floating centered on the full-viewport gradient**. No
chrome, no header bar, no nav ‚Äî the gradient canvas *is* the design (¬ß6: *"the gradient is part of the
design, don't crowd it"*).

| Element | Spec | Source |
|---|---|---|
| **Canvas** | Full-viewport `--bg-gradient`; the calculator centered both axes via flexbox, `min-height: 100vh`; generous breathing room around the glass. | ¬ß6, UR-030 |
| **Calculator body** | A **single** glass panel. `max-width: 360px` on desktop; full-width with `16px` page margin on phone. Internal padding `20‚Äì24px`. `border-radius: 24px`. Glass recipe ¬ß5 (body blur `16px`). | ¬ß6, ¬ß5, UR-031 |
| **Stacking inside the body** | Top‚Üíbottom, single column: **(1)** display sub-panel, **(2)** keypad grid. Nothing else in M2. (The toggle slot and history slot are *reserved*, not rendered ‚Äî ¬ß5 below.) | ¬ß6, F3, F4 |
| **Glass recipe (every pane)** | `background: var(--glass-fill)` (display uses `--glass-fill-display`); `backdrop-filter: blur(‚Ä¶) saturate(140%)` (+ `-webkit-` twin for Safari); `border: 1px solid var(--glass-border)`; two-layer elevation (outer drop `0 8px 32px rgba(0,0,0,.18)` + inner top sheen `inset 0 1px 0 rgba(255,255,255,.25)`). **Fallback** when `backdrop-filter` unsupported: higher-alpha solid-ish fill (¬ß5) so text stays legible ‚Äî never transparent-with-no-blur. | ¬ß5 |

The body is the "premium glass object." Single-panel center-stage is the deliberate choice (validated
interpretation): it maximizes the glass-object feel without adding structural complexity (CP-13
anti-over-engineering ‚Äî the Domain Lens makes design-craft the one real axis, not feature count).

## 2. Display sub-panel ‚Äî two lines, top of body

A glass sub-panel at the top of the body, reading as a **recessed window** (intentionally *more*
transparent than the body ‚Äî `--glass-fill-display`, ¬ß5/¬ß6).

| Property | Spec | Source |
|---|---|---|
| **Glass** | `--glass-fill-display` fill, blur `16px` saturate `140%`, `--glass-border` edge, `border-radius: 20px`, two-layer elevation. | ¬ß5, ¬ß6 |
| **Geometry** | Min-height `~96px`; internal padding `16‚Äì20px`; **right-aligned**, content bottom-aligned (a calculator reads from the bottom-right). | ¬ß6, UR-003 |
| **Live region** | The display container carries `role="status"` + `aria-live="polite"` so screen readers announce every result and error change. | UR-023, ¬ß8 |
| **Numeric rendering** | `font-variant-numeric: tabular-nums` on both lines so digits don't jitter as the value changes. | UR-004, ¬ß4 |

**Two lines, top‚Üíbottom:**

1. **Pending-expression line (secondary, smaller, on top).** Type: Inter `0.875rem`, weight `400`,
   color `--text-secondary`, tabular-nums. Content = `pendingLine(state)` = `<accumulator> <glyph>`
   (e.g. `12 √ó`), the glyph being the true-Unicode operator (UR-014). **Shown only when** `accumulator
   !== null` AND `pendingOperator !== null` AND `errorState === null` AND `justEvaluated === false`
   (UR-008, D-M2-DM-03); **hidden** (collapsed, no layout jump) otherwise. The display reserves its
   vertical space so showing/hiding the line does **not** shove the big number around.
   *(Source: UR-006/007/008, ¬ß4, INT-3.)*

2. **Primary readout (large, below).** Type: Inter, `clamp(2.5rem, 12vw, 4rem)`, weight `300` (light),
   color `--text-primary`, tabular-nums, right-aligned. Content = the tag‚Üísentence mapping over
   `getDisplayValue(state)` (UR-001/002):
   - numeric string ‚Üí render verbatim (e.g. `42`, `-2`, `3.5`);
   - `'divide-by-zero'` ‚Üí **`Cannot divide by zero`**;
   - `'overflow'` ‚Üí **`Number too large`**.

   > **Error-sentence rendering note (INT-2, UR-002, UR-016).** When the readout holds an error
   > *sentence* (not a number), it is multi-word text in a slot sized for a big number. The mockups
   > must show the error state gracefully: the sentence **wraps** (or scales down) inside the display
   > rather than overflowing the glass ‚Äî e.g. allow the readout to drop to a smaller size / wrap to 2
   > lines in error mode. The error is conveyed by the **sentence text** (`--error` color *plus* the
   > words), never by color alone (UR-016, UR-026, ¬ß8).

**The four display conditions the mockups must each depict** (UR-005) ‚Äî fresh `0`, mid-entry (e.g.
`3.5`), result (post-equals value), error (a sentence). See ¬ß4 for the full per-state visual table.

**Spanning decisions (concrete, locked for the conservative mockup):**
- **`=` spans 2 rows** in the right-most column (rows 4‚Äì5), giving equals a tall, prominent target
  consistent with its `--accent` emphasis. *(F8 permits equals to span; a tall right-column equals is
  the cleaner fit once `¬±`/`%` are removed ‚Äî it absorbs the freed row-5 right cell.)*
- **`0` spans 2 columns** (row 5, cols 1‚Äì2), the conventional wide-zero. Hero precedent (`.btn-zero {
  grid-column: span 2 }`). Left-aligned label with `padding-left` to echo the digit column above it.
- All other cells are single 1√ó1.

**Why operators run down the right column (rows 1‚Äì4) rather than the hero's mixed placement:** with
`¬±`/`%` gone, a clean right-hand operator column (`√∑ √ó ‚àí +`) is the standard, most-scannable 4-function
arrangement and leaves row 1's left two cells for the two clears (AC, CE). This is pure UX craft
(CP-7); the owner picks a *mockup*, not this grid (OQ3).

**Glyph rule (UR-014, F10):** operator button faces render the **true-Unicode** glyph `√∑ √ó ‚àí +`
(U+00F7 / U+00D7 / U+2212 / U+002B), **never** the ASCII `/ * - +`. Same glyphs appear in the pending
line (UR-007).

**Button type styling (¬ß6, F8, UR-024):**
- **Digits + `.`** ‚Äî `--glass-fill` glass pane, label `--text-primary`, Inter `1.5rem`/weight `500`.
- **Operators `√∑ √ó ‚àí +`** ‚Äî `--operator-accent` (coral) as the distinguishing fill/label; glyph **plus**
  fill together identify an operator so the cue survives grayscale/color-blindness (UR-024, ¬ß8).
- **Equals `=`** ‚Äî `--accent` (indigo) fill, high-emphasis (it's the commit action).
- **Clears AC/CE** ‚Äî util styling, de-emphasized label color (`--text-secondary`), so they read as
  secondary to the digits/operators.

## 4. States ‚Äî per-state visual spec

Every state below is a **derivation of the single held `EngineState`** (data-model ¬ß2/¬ß3) ‚Äî M2 stores
no "which state" flag; each is read at render. The mockups should depict the starred (‚òÖ) states.

### 4.1 Display states (UR-005)

| State | Trigger | Primary readout | Pending line | Notes |
|---|---|---|---|---|
| **Fresh ‚òÖ** | load / after AC | `0` | hidden | The default screen the first mockup frame shows. |
| **Mid-entry ‚òÖ** | user typing | live `entryBuffer` (e.g. `3.5`) | shown if an op is pending (`12 √ó`), else hidden | tabular-nums keeps it steady (UR-004). |
| **Result ‚òÖ** | post-`=` (`justEvaluated`) | resolved value | **hidden** (cleared on equals) | fires the result motion (¬ß4.3). |
| **Error ‚òÖ** | `errorState !== null` | the **sentence** (`Cannot divide by zero` / `Number too large`) | **hidden** | error styling on; see ¬ß4.2. |

### 4.2 Error state (UR-016, UR-017, INT-6)

- Primary readout shows the **sentence**, color `--error`, **plus** the words (not color alone ‚Äî UR-026).
- Sentence **wraps / scales** inside the display glass (¬ß2 note) ‚Äî never overflows the pane.
- M2 does **not** disable digit/operator buttons (M1 no-ops them ‚Äî UR-017); but **CE and AC stay
  visually active and obviously reachable** so the escape is legible (INT-6). The mockup may *gently
  de-emphasize* the inert keys and/or *emphasize* CE/AC during error ‚Äî optional craft, but the escape
  must never be ambiguous.
- The latch clears by derivation on the next render after CE/AC (UR-018) ‚Äî no manual error flag.

### 4.3 Interaction states (per button)

| State | Visual | Source |
|---|---|---|
| **:hover** (desktop) | subtle fill-alpha increase + faint glow, `150ms ease`. | ¬ß7 |
| **:active** (press) | scale `0.96` + brief fill-brightness lift, `transition: transform 80ms ease-out, background 120ms`. Springs back, no bounce. Click-only (a keyboard press updates the display without the press flash ‚Äî `02-flows.md` ¬ß1). | ¬ß7, UR-025 |
| **:focus / :focus-visible** | **visible focus ring `3px solid var(--accent)` with `3px` offset** (v3 baseline; exceeds ¬ß8 2px floor) ‚Äî `outline:none` is never used without this replacement. | UR-020, ¬ß8 |
| **Result motion** (display, on `=`) | new value fades+rises: `opacity 0‚Üí1` + `translateY(6px‚Üí0)` over `200ms ease-out`. The *one* place the result motion fires (operators use press motion only ‚Äî `02-flows.md` ¬ß3/¬ß4). | ¬ß7 |
| **Reduced motion** | under `@media (prefers-reduced-motion: reduce)`, drop transforms/translates (press-scale, result fade+rise) ‚Üí instant state changes. **Not optional.** | UR-025, ¬ß7, ¬ß8 |

Focus, press, and reduced-motion are **DOM/CSS-native** ‚Äî no JS view state (data-model ¬ß2).

## 5. Responsive layout + reserved slots

| Breakpoint | Layout | Source |
|---|---|---|
| **Desktop** (‚â≥640px) | calculator body centered, `max-width: 360px`. | ¬ß6, UR-031 |
| **Phone** (‚â≤640px) | body full-width with `16px` page margin; same single-column stack (display over keypad). 4-col grid holds; buttons stay `‚â•64px` and unclipped. | ¬ß6, UR-031 |
| **Reflow** | no horizontal scroll, no clipped buttons at `~375px` or `~1280px`. Grid is `repeat(4, 1fr)` so columns fluid-resize between the two sizes. | UR-031 |

**Reserved slots ‚Äî lay out so M3/M4 drop in without restructuring (F3, F4; do NOT build them):**

- **History slot (M3, F3, UR-032).** The calculator body must sit in a layout that **leaves a sibling
  slot** for the history panel ‚Äî **desktop: side-by-side** room (history to the side, ~240px,
  switching under ~640px); **phone: stacked** room (history above/below). Concretely: wrap the body in
  a flex container (row on desktop, column on phone) rather than hard-centering a lone panel with no
  room. **M2 renders no history markup** ‚Äî it only must not preclude M3. *(The hero's `.app-layout`
  flex-row with a 240px aside is the precedent shape; M2 ships only the calculator child of it.)*
- **Theme-toggle slot (M4, F4).** Reserve the top-right corner inside the calculator body for M4's
  toggle (hero placed a 36√ó20px pill at `top:16px; right:16px`). **M2 renders no toggle** ‚Äî just don't
  occupy that corner with anything M2 needs back. M2 ships **one (default/light) theme**, all theme
  colors via `var(--token)` (UR-029) so M4 swaps the set without touching M2 markup.

## 7. The 3-variant mockup brief (Step 5.5 targets ‚Äî owner's "push to the limit" steer)

The owner locked a wide-exploration steer (*"DORONG SAMPAI MENTOK"*). The three generated mockups must
be **distinctly different targets**, not three near-copies. Each builds the **same 15-key surface** (¬ß3)
and honors **all ¬ß6 accessibility callouts** (the craft floor never dials, even for the wildcard) ‚Äî they
differ in how far they stray from the locked tokens/proportions.

- **v1 ‚Äî Conservative (strict baseline).** Build the spec **verbatim**: locked tokens (¬ß3 palette),
  the exact ¬ß3 grid layout (AC/CE/√∑/√ó row 1; right-column operators; `0` spans 2 cols; `=` spans 2
  rows), glass recipe ¬ß5 as-written (body blur 16px, button blur 8‚Äì10px), motion ¬ß7 as-tabled. This is
  the "design-system is law" reference ‚Äî the safe floor the owner can always fall back to. Zero token
  drift.

- **v2 ‚Äî Bold (explore freely *within* glassmorphism + core tokens).** Keep the locked palette tokens
  and the glassmorphism identity, but push **proportion, blur, accent emphasis, and layout** hard:
  e.g. a larger/heavier display, stronger blur + saturation, a more dominant equals, alternative (but
  still 4-col) key arrangement, bolder operator-coral presence, more pronounced elevation/sheen. Stays
  *recognizably the same product* ‚Äî same tokens, same glass ‚Äî but turned up. Probes "how expressive can
  we get without leaving the system."

- **v3 ‚Äî Wildcard (probe the upper bound ‚Äî MAY cross tokens).** Permitted to **cross the locked
  tokens** to find the ceiling: a different gradient (e.g. a richer/multi-stop or animated canvas),
  extra effects (layered glass, inner colored glow, refraction/noise texture, a more daring accent
  pairing), more dramatic typography or an asymmetric composition. Still a glassmorphism calculator and
  still the 15-key surface ‚Äî but unapologetically *the maximalist take*. This is the "what if we went
  all the way" frame the owner asked for. **The one hard floor it may NOT cross: ¬ß6 accessibility**
  (contrast AA, real buttons, focus ring, reduced-motion, live region) ‚Äî daring visuals, but still
  usable.

> Owner picks one mockup (or a hybrid) ‚Üí captured in `_picked.md`. The pick is a **taste** decision
> (CP-7 human-owned aesthetic); this brief arms all three generators with clear, divergent targets so
> the choice is real, not three shades of the same.
