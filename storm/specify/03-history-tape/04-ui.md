# M3 UI Spec — History Tape Panel (build-ready brief for the 3 mockup variants)

> **What this file is.** The concrete, build-ready UI specification for the **history tape panel** — the
> glass aside that sits beside the v3 calculator slab (desktop) / stacked with it (phone), rendering the
> in-memory `HistoryEntry[]` (`01-data-model.md`). It is a **SPEC, not code** — every visual ties to a
> design-system `§N` or a briefing fact (`F-M3-*`). The **3 mockup variants (§9) are the code**; this file
> is the brief they are generated from, and the owner picks the resonant one (validation is visual).
>
> **Scope boundary.** M3 owns the tape's **content + panel UI + its own state list + the recording seam**.
> **Placement is design `§6` (F-M3-5), inherited verbatim — not re-derived here.** Token values are cited
> **once** (§0) and never restated per-state (density-at-source, F-M3-6 / #FF-027). M2 (the keypad), M1
> (the engine), and the hook *mechanism* (`06-tech-choices.md`) are out of scope (§8 ownership map).

## 1. Overall composition — a glass aside that belongs to the slab

The tape is **one glass panel**, applying the same recipe (§0 → design §5) as the calculator body, so it
reads as the *same family of object* — a second pane of the same frosted glass floating on the same aurora
canvas. It is **not** chrome, **not** a modal, **not** a drawer: it is a quiet sibling surface.

| Element | Spec | Source |
|---|---|---|
| **Panel** | Single glass pane. **Desktop:** the existing `.history-slot` aside, ~240px wide, `align-self: stretch` (matches slab height), scrollable internal list. **Phone:** stacked, full-width, capped height ~140px, scrollable. | §0, F-M3-5, F-M3-8 |
| **Glass recipe** | Inherited verbatim from design §5 — same fill, blur+saturate, 1px `--glass-border`, two-layer elevation, `backdrop-filter` fallback. The tape may read **slightly more recessed** (like the display sub-panel) so it doesn't compete with the keypad for emphasis — a craft call left to the mockups (§9), not a token change. | §5, §0 |
| **Internal structure** | top→bottom: an **optional quiet label/heading** ("History" — see §3 a11y) then the **entry list** (the scroll region). Nothing else — no toolbar, no clear button (§5), no filter, no search. | F-M3-5 |
| **Header room** | The slab already reserves a top-right toggle corner for M4; the tape panel is independent of that — M3 adds no controls to the slab. | F-M3-8 |

The tape is a **read-only display surface** (F-M3-3): no entry is interactive, nothing is clickable,
nothing is focusable inside the list except the scroll region itself when it overflows (§6). This is the
deliberate minimalism the zero-stakes scope asks for (CP-13 anti-over-engineering).

## 2. Entry anatomy — the two-string `HistoryEntry`

Each entry renders the two display-ready strings from `01-data-model.md` (`{ expression, result, id }`):
**expression** (e.g. `"12 + 3"`, `--text-secondary`) and **result** (e.g. `"15"`, `--text-primary`),
both Inter `0.875rem`/400/`tabular-nums`, **right-aligned** (a calculator reads from the right; matches
M2's right-aligned readout). The `id` is the render key only — never displayed (data-model §1).

> **OQ-M3-2 resolved — primary layout: TWO lines per entry (expression above, result below).**
>
> **Pick: two-line.** Expression on top in `--text-secondary`; the result on the line below in
> `--text-primary`, prefixed with the `=` glyph (`= 15`) so the relationship reads without a single-line
> equation. Rationale (CP-7 UX-craft, AI-autonomous):
> - **Fits the narrow ~240px / ~140px panel without crowding.** A single-line `12 + 3 = 15` competes for
>   horizontal room that the panel barely has; two lines use the *vertical* axis the scroll already owns,
>   so long values stay legible (feeds OQ-M3-5, §4).
> - **Honors the design-system typography split cleanly.** `§4`/F-M3-6 colors expression-secondary /
>   result-primary; stacking them makes the **result the visually dominant line** (primary color, the
>   thing the user glances back for) with the expression as quiet context above it — the natural reading
>   hierarchy for "what did that one come out to."
> - **The data model supports either** (data-model §3: two separate string fields, `expression` need not
>   carry the `=`). So this is a *render* choice, reversible at mockup time.
>
> **The 3 mockups WILL explore the single-line alternative** (`12 + 3 = 15` on one row, §9 Conservative) so
> the owner sees both and picks. The two-line layout above is the spec's default target; the pick is visual.

**Entry spacing/dividers:** entries are separated by quiet vertical rhythm — a small gap and/or a
hairline `--glass-border`-alpha divider between rows (a craft call for the mockups, §9). No heavy rules, no
zebra striping — the calm-glass feel (design §7 "calm, not a toy") governs.

## 3. Per-state visual spec

The tape has four visual states, all **derivations of the `HistoryEntry[]` length + the just-appended
beat** — M3 stores no "which state" flag (mirrors M2's derive-at-render discipline, `02-calculator-ui/04-ui.md §4`).

### 3.1 Empty tape (no calculations yet)

The default state at load and after AC (data-model §4). **Pick: a quiet placeholder, not a blank void.**

- A single centered, low-emphasis line in `--text-secondary` — e.g. *"No calculations yet"* (or a softer
  *"Your history appears here"*) — Inter `0.875rem`/400, dimmed. **Reason:** v2 Bold was *not* picked partly
  because "the M3 history slot read as **empty void**" in a standalone frame (`_picked.md` rejected-table);
  a quiet placeholder fills that void gracefully and tells the first-time user what the panel is *for*. It
  is **not** an illustration, not an icon, not a CTA — one calm sentence (CP-13 anti-inflation).
- The placeholder is **decorative/explanatory**, marked so it is not announced as a list item (it is not an
  entry); it disappears the instant the first entry lands.

### 3.2 Populated (entries list)

- The entry list (§2) renders newest-at-the-visible-edge (F-M3-5). Data is oldest-first in the array
  (data-model §2 D-M3-DM-01); the **render** places newest at the top edge (e.g. `flex-direction:
  column-reverse` or a reversed read) — a render concern, exactly as the data model deferred it here.
- The list fills the panel top-down from the newest entry; older entries recede below (desktop) / are
  scrolled away (phone capped height).

### 3.3 Scrolled (overflow)

- When entries exceed the panel height (desktop ~slab-height; phone ~140px), the **list scrolls** —
  unbounded tape, scroll absorbs growth (data-model §5 D-M3-DM-03; no eviction, no data loss).
- **Native scroll**, vertical only. A subtle top/bottom fade-mask (gradient `mask-image`) may hint
  more-content-above/below without a heavy scrollbar — a craft option for the mockups (§9), not required.
- The scroll region is **keyboard-reachable when it overflows** (§6) so keyboard users can read the whole
  tape — but it is **never a focus trap** (§6).

### 3.4 Just-added (the slide-in beat)

- On a genuine equals-resolve (the INT-M3-1 append, data-model §4), the **new line slides in from the top
  edge, `180ms ease-out`**, gently pushing older entries down — design §7 / **F-M3-7**, inherited
  verbatim. This is the tape's one motion; it echoes M2's result fade+rise (the same "this updated"
  register, design §7) so the two surfaces feel coordinated.
- **`prefers-reduced-motion: reduce` → the slide-in is dropped** (instant insert, no transform). Not
  optional (F-M3-7, design §8, craft floor C3).
- The motion is **CSS-native** (a keyframe/transition on insert); no JS animation loop (mirrors M2's
  DOM/CSS-native motion, `02-calculator-ui/04-ui.md §4.3`).

## 4. Long expression / result inside the narrow panel (OQ-M3-5)

> **OQ-M3-5 resolved — WRAP, never truncate (no data loss).** A long expression or a long result (e.g.
> `9999999 × 8888888` → a many-digit result) inside the ~240px / ~140px panel **wraps to the next line**;
> it is **never clipped, ellipsed, or hidden**.
>
> **Pick: wrap.** Rationale (CP-7 technical, AI-autonomous):
> - **Mirrors M2's no-truncation trust principle (D-006).** M2's readout shrinks→scrolls but **never
>   truncates** — the user must always be able to read the full value. The tape is a *record*; silently
>   dropping digits would betray the same trust the whole product honors. The two-line layout (§2) already
>   gives each entry its own vertical room, so wrapping a long result onto a second/third sub-line is
>   natural and cheap.
> - **Wrap over per-line horizontal scroll.** Per-entry horizontal scroll (an alternative) hides data
>   behind a gesture and is awkward on the narrow phone strip; vertical wrap keeps everything visible at a
>   glance, and the panel's vertical scroll (§3.3) already absorbs the extra height. `tabular-nums` +
>   `overflow-wrap: anywhere` (or `break-word`) on the entry text lets a long unbroken number wrap mid-run
>   rather than forcing horizontal overflow.
> - **Numbers stay grouped where possible.** Wrapping is the floor (no data loss); the mockups may refine
>   *where* a long number breaks for legibility (§9) — but the invariant is **all characters always
>   visible**.
>
> **`05-edge-cases.md` refines this** — exact break behavior for pathological values, the interaction with
> the slide-in motion on a multi-line entry, and the chained-calc framing (INT-M3-4) are its job. This file
> pins the principle: **wrap, never lose data.**

## 5. Clear affordance (OQ-M3-1)

> **OQ-M3-1 resolved — AC clears the tape; NO dedicated "clear history" control.**
>
> **Pick: AC is the clear, no separate button.** The module baseline says the tape "clears with a full
> **AC** / tab close" (`03-modules.md:82`); the data model confirms AC empties the array (data-model §4).
> Rationale (CP-13 anti-inflation + profile-fit):
> - **AC already means "reset everything."** A user reaching for All-Clear expects a clean slate; clearing
>   the tape with it is the least-surprising behavior and needs **zero new UI**. A dedicated control adds a
>   button, an `aria-label`, a focus stop, a confirm-or-not decision, and a visual element competing with
>   the calm tape — bespoke surface for a nicety the zero-stakes scope doesn't ask for (least-bespoke-code,
>   F-M3-10).
> - **CE does NOT clear the tape** (data-model §4) — only AC. The tape outlives entry-level clears; it dies
>   only on AC or tab close (F-M3-2).
>
> **Where any clear appears in the UI: nowhere new.** The clear is triggered by the *existing* AC key on the
> keypad (M2's surface). The tape panel renders **no clear affordance of its own** — on AC it simply returns
> to its empty-state placeholder (§3.1). If a future owner asks for a dedicated control, it would live as a
> small de-emphasized button in the tape header running the same `tape = []` clear — but that is **out of
> scope now** and explicitly not built (parked-by-omission, not designed).

**Note — read-only is total (F-M3-3, OQ-M3-3 confirmed OUT):** no entry is clickable, no click-to-reuse, no
copy, no export, no per-entry delete. The only mutation paths are *append* (on equals-resolve) and *clear-all*
(on AC). State the boundary in the mockups: entries have **no hover-affordance suggesting interactivity**
(no pointer cursor, no hover-lift) — they are text, not buttons.

## 6. Accessibility callouts (craft floor C3 — non-negotiable, design §8)

The tape is **small, but the craft floor holds at full height** (C3 — small scope, same bar). The mockups
MUST honor these:

| Callout | Requirement | Source |
|---|---|---|
| **Remove `aria-hidden`** | The reserved slot ships `aria-hidden="true"` (F-M3-8, `index.html:28`). M3 **removes it** so the tape is exposed to assistive tech — leaving it hidden would make the whole feature invisible to SR users. | F-M3-8 |
| **Semantic list markup** | The entry list is a **real list** — `<ul>` / `<li>` (or `role="list"`/`role="listitem"`), so SR users hear "list, N items" and can navigate item-by-item. Each `<li>` is one `HistoryEntry`. **Not** a pile of `<div>`s. | design §8 (semantics), C3 |
| **Region label** | The panel is a labeled region — `aria-label="Calculation history"` (or an `<h2 class="sr-only">History</h2>` heading the list references), so SR users know what the region is. If a visible header is shown (§1), it doubles as the label. | design §8 |
| **Not a focus trap** | The tape is **read-only** — it must **never** trap focus. Entries are **not** focusable (they are text, not controls — §5). The **only** focusable element is the **scroll container when it overflows** (`tabindex="0"` *conditionally*, mirroring M2's REVIEW fix where the scroll region got a conditional `tabindex` so keyboard users can scroll without a permanent dead tab-stop). When the list fits without scroll, no tab stop is added. | design §8, M2 REVIEW precedent |
| **Reduced motion** | `prefers-reduced-motion: reduce` **drops the slide-in** (§3.4) → instant insert. | F-M3-7, design §8 |
| **Contrast over glass** | `--text-secondary` (expression) and `--text-primary` (result) must clear **WCAG AA** over the actual glass-over-aurora backdrop (≥4.5:1 normal text). `--text-secondary` is the at-risk one (it's dimmed by design); if a spot fails, **raise the panel alpha (more opaque glass) before darkening text** — the mandated order from M2 (`02-calculator-ui/04-ui.md §6`, design §8). The v3 baseline uses a near-opaque scrim trick for its readout (~17:1, `_picked.md`); the tape can borrow the same recessed-scrim approach if the dimmed secondary text needs it. **Verify against screenshots, both themes.** | design §8, F7, M2 precedent |
| **No color-alone** | The expression/result distinction is conveyed by **position + the `=` prefix on the result line** (§2), not by color alone — so the hierarchy survives grayscale / color-blindness. | design §8 (no color-alone) |
| **`aria-live` — NOT on the tape** | The display already carries `role="status"`/`aria-live="polite"` (M2, `index.html:40`) and announces the result when equals resolves. The tape should **not** *also* be a live region — that would double-announce every result (the display says "15", then the tape says "12 + 3 = 15" again). The tape is a **static, navigable** record SR users read on demand; the display owns the live announcement. *(Craft call — avoids the double-announce a11y anti-pattern.)* | design §8, M2 live-region |

## 9. The 3-variant mockup brief (mockup targets)

The owner validates **visually** — three distinctly different tape treatments, generated next (orchestrator
dispatches them separately; **this file does not generate them**). **All three are hard-anchored:** they
keep the **v3 aurora-glass DNA** (frosted translucent panes belonging to the picked baseline,
`_picked.md`), render the **same two-string `HistoryEntry`** (§2), populate the **existing 240px slot**
(§0, F-M3-8), and honor **every §6 accessibility callout** (the craft floor never dials — semantic list, no
focus trap, reduced-motion, AA contrast, no data loss). They **differ on entry layout density, panel
treatment, and the add-motion character**:

- **v1 — Conservative (strict baseline).** The spec **verbatim** but probing the **single-line** entry
  alternative to OQ-M3-2: each entry on one row `12 + 3 = 15` (expression `--text-secondary`, result
  `--text-primary`, the `=` between), compact rows, a hairline divider, the §7 slide-in exactly as tabled
  (`180ms ease-out`). Glass recipe §5 as-written, recessed like the display. The calm, dense, token-faithful
  reference — the safe floor. *(Explores the single-line layout the spec's two-line default did not pick, so
  the owner sees both — §2.)*

- **v2 — Bold (two-line, expressive within the system).** The spec's **two-line** default (§2) turned up:
  larger result line, stronger primary/secondary contrast, more generous per-entry spacing, a more
  pronounced recessed-glass panel (deeper blur / clearer inner sheen), the empty-state placeholder given a
  bit more presence (still one calm line, §3.1). The slide-in gets a touch more travel/ease character (still
  `180ms`, still reduced-motion-safe). Stays *recognizably the same product* — same tokens, same glass —
  but the tape feels deliberately crafted, not an afterthought. Probes "how premium can the record feel
  without leaving the system."

- **v3 — Wildcard (maximalist tape — MAY push panel treatment + motion).** The upper-bound take, matching
  the energy of the picked v3 calculator: a more dramatic panel treatment (layered glass, a subtle inner
  colored glow echoing the aurora, a top/bottom fade-mask on the scroll, §3.3), the newest entry possibly
  given a brief emphasis-beat as it lands (a momentary highlight that settles), denser-yet-legible typographic
  rhythm. The add-motion is the most characterful here (still `180ms ease-out` slide from the top, still
  instant under reduced-motion). Still a glassmorphism tape, still the two-string read-only list — but the
  maximalist "this record feels alive" frame. **The one hard floor it may NOT cross: §6 accessibility**
  (semantic list, no focus trap, reduced-motion drops all motion, AA contrast over glass, no data loss) —
  daring visuals, still fully usable.

> Owner picks one (or a hybrid) → captured in M3's `_picked.md`. The pick is a **taste** decision (CP-7
> human-owned aesthetic); this brief arms all three with clear, divergent targets — single-line vs two-line
> entry density, restrained vs maximalist panel, calm vs characterful add-motion — so the choice is real,
> not three shades of the same.
