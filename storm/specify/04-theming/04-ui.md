# M4 Theming — UI Concern: Toggle Control + Light Palette

> Owner-validated UX interpretation (inline, this session): a small icon-button toggle
> (sun ☀ / moon ☾) in the reserved `.toggle-slot` corner of the calculator body — single
> `<button>` that flips light↔dark on click; 300ms cross-fade on activation; full a11y.

---

## 1. Toggle control — anatomy

### 1.1 Element

A single `<button>` element placed inside the existing `.toggle-slot` div
(`index.html:48`). **Not** a checkbox, not a styled `<div>` — a real `<button>` is
keyboard-operable (Enter / Space) and announced by screen readers without ARIA hacks.

```html
<!-- replaces the aria-hidden="true" on .toggle-slot once M4 fills it -->
<button
  id="theme-toggle"
  class="theme-toggle"
  aria-label="Switch to light theme"
  aria-pressed="false"
>
  <!-- icon rendered via CSS content or inline SVG — see §1.3 -->
</button>
```

`.toggle-slot` loses `aria-hidden="true"` when filled; the button becomes the interactive
landmark.

### 1.2 Icon — sun / moon, current-theme convention

The icon shows **the currently active theme**, not the target theme:

- **Dark active** → moon icon `☾` (or crescent SVG)
- **Light active** → sun icon `☀` (or sunburst SVG)

Rationale: "you are here" orientation — the icon mirrors the visible state, so users
always know which theme is on. This is the conventional iOS/macOS approach (the status
icon, not the action icon).

`aria-label` is dynamic and names the **action** (what will happen on press), not the
state — this is the correct screen-reader pattern:

- Dark active → `aria-label="Switch to light theme"`
- Light active → `aria-label="Switch to dark theme"`

`aria-pressed` reflects the **current active state** — set `"true"` when light is active,
`"false"` when dark is active (treating "light = pressed/on" as the toggle's on-state, so
the button's pressed state is unambiguous). BUILD may choose the opposite polarity — the
constraint is that `aria-pressed` + `aria-label` are **always consistent** and updated
together on every toggle.

### 1.3 Icon implementation

Prefer **inline SVG** (no font-icon dependency, scales without CORS, colour inherits via
`currentColor`). Acceptable fallback: CSS-`content` Unicode characters (☀ / ☾) if SVG
adds friction in BUILD. Either approach must produce a crisp, legible icon at the button's
rendered size.

Icon colour: inherits `currentColor` from the button (matches `--text-primary` or
`--text-on-scrim` depending on slot placement — confirm in BUILD against the glass).

### 1.4 Size and placement within the slot

- Button: `32×32px` minimum (exceeds §8 44px tap-target floor when accounting for the
  slot's own clickable area; pad the button to `44×44px` minimum touch target with CSS
  `padding` if needed).
- Icon: `20×20px` inside the button, centered.
- Placement: the slot sits in the top corner of `.calc` per `layout.css:135` — the button
  fills the slot flush (no additional margin that would shrink the tap area).

### 1.5 Press feedback — design-system §7

On `:active`, apply the standard keypad press treatment scaled to the icon-button context:

```css
.theme-toggle:active {
  transform: scale(0.92);
  transition: transform var(--motion-press); /* 80ms ease-out */
}
```

No bounce. Quick, tactile, springs back — consistent with the keypad feel (§7).

### 1.6 Theme cross-fade — 300ms

On toggle activation, the `data-theme` attribute on `document.documentElement` flips
(per `03-rules.md` R-M4-05). The CSS transition on the token swap is a **300ms cross-fade**
across all themed `background`, `color`, and `border` properties (§7):

```css
:root {
  transition:
    background 300ms ease,
    color 300ms ease,
    border-color 300ms ease;
}
```

Under `prefers-reduced-motion: reduce` (R-M4-08): the transition is dropped to instant —
remove the `transition` declarations via a `@media (prefers-reduced-motion: reduce)` block.
Theme swap still happens; only the animation is suppressed.

## 2. Accessibility

| Requirement | Spec |
|---|---|
| Element | Real `<button>` (keyboard-operable: Enter, Space) |
| `aria-label` | Dynamic; names the **action** ("Switch to light/dark theme") |
| `aria-pressed` | Reflects active state; updated on every toggle (consistent with `aria-label`) |
| Focus ring | `3px solid var(--accent)` / `3px offset` — the shipped v3 baseline (exceeds §8 floor of 2px/2px) |
| Min touch target | `44×44px` (§8 floor; achieved via padding on the 32×32 button) |
| Reduced motion | Cross-fade dropped to instant via `@media (prefers-reduced-motion: reduce)` (R-M4-08) |
| Screen reader | Icon is decorative (`aria-hidden="true"` on the SVG/`<span>`); the button's `aria-label` is the full announcement |
| Color independence | Icon shape (sun vs moon glyph) distinguishes themes — not color alone |

The focus ring uses `var(--accent)` which is defined in **both** light and dark token sets
(see §3 below) and must remain visually distinct on both glass surfaces.

## 3. Light theme palette — mirror-vibrant dark

> This supersedes the stale §3.1 pastel palette in `08-design-system.md` for M4 purposes.
> The §3.1 table predates the shipped v3 Wildcard pick (D-003, CF-1, CF-4). The light theme
> derives from the **shipped** `tokens.css` dark baseline, not from §3.1. (Cited: CF-4,
> `03-rules.md` R-M4-06.)

The light palette is the **bright sibling of the v3 aurora dark**: vibrant-but-light
gradient (bright blue/violet/pink aurora), white translucent glass (higher fill alpha than
dark), indigo `#7c5cff`-family + coral `#ff5e7a`-family accents still popping, dark text on
light glass. "Mirror vibrant" means the chromatic energy of the dark aurora is preserved —
lifted into the light range, not diluted into pastels (TD-1, `_briefing.md`).

### 3.1 Candidate token values (starting point — tunable in BUILD)

These are concrete starting values for the BUILD mockups and are **not final** — BUILD
adjusts them to hit contrast targets and achieve the vibrant-light aesthetic in-browser.

```css
[data-theme="light"] {
  /* Accent system — same family as dark, adjusted for light-bg legibility */
  --accent:                    #5b3de8;   /* indigo, darkened vs dark's #7c5cff for AA on light */
  --accent-glow:               rgba(91, 61, 232, 0.40);
  --accent-dark:               #4a2ec8;
  --accent-hover-from:         #6a4df5;
  --accent-hover-to:           #5030d0;
  --operator-accent:           #e8285a;   /* coral-rose, darkened vs dark's #ff5e7a for AA on light */
  --operator-accent-dark:      #cc1f4a;
  --operator-accent-hover-from:#f03868;
  --operator-accent-hover-to:  #d42050;
  --operator-glow:             rgba(232, 40, 90, 0.35);
  --text-on-accent:            #ffffff;   /* white text on accent-filled keys — unchanged */
  --error:                     #c0122a;   /* darker red for AA on light glass */

  /* RGB triplets — update to match new accent values */
  --accent-rgb:           91, 61, 232;
  --operator-accent-rgb:  232, 40, 90;
  --scrim-rgb:            240, 238, 255; /* light scrim — near-white tinted indigo */
  --calc-fallback-rgb:    230, 228, 255;
  --op-shadow-rgb:        180, 0, 60;
  --eq-shadow-rgb:        50, 20, 160;

  /* Text — dark on light glass for AA contrast */
  --text-on-scrim:        #1a1040;       /* dark indigo on near-white scrim — high contrast */
  --text-on-scrim-soft:   rgba(26, 16, 64, 0.65);
  --text-primary:         #1a1040;       /* dark indigo on light glass (digit labels) */
  --text-secondary:       rgba(26, 16, 64, 0.60);

  /* Glass — white translucent, higher alpha than dark (more visible on light gradient) */
  --glass-fill:          rgba(255, 255, 255, 0.52);
  --glass-fill-hover:    rgba(255, 255, 255, 0.65);
  --glass-fill-display:  rgba(255, 255, 255, 0.38);
  --glass-border:        rgba(255, 255, 255, 0.80);
  --glass-border-bright: rgba(255, 255, 255, 0.95);

  /* Display scrim — near-white tinted indigo (flips the dark scrim) */
  --display-scrim:       rgba(240, 238, 255, 0.82);

  /* Background — bright blue/violet/pink aurora (mirror of dark aurora's hue family) */
  --bg-base:   #e8e4ff;
  --bg-aurora:
    radial-gradient(40% 50% at 20% 25%, #a78bfa 0%, transparent 60%),
    radial-gradient(45% 55% at 80% 20%, #f472b6 0%, transparent 60%),
    radial-gradient(50% 60% at 75% 80%, #60a5fa 0%, transparent 62%),
    radial-gradient(45% 50% at 25% 80%, #a855f7 0%, transparent 60%),
    linear-gradient(135deg, #ddd6fe 0%, #f3e8ff 50%, #fce7f3 100%);
  --bg-blob:   radial-gradient(circle, rgba(232, 40, 90, 0.35) 0%, rgba(91, 61, 232, 0.28) 45%, transparent 70%);

  /* Motion tokens — unchanged from dark (same timing, same feel) */
  /* --motion-* and --glass-glow-hover are not theme-semantic; leave at :root values */

  /* Glass-recipe structural tokens — unchanged (not theme-semantic per T-122) */
  /* --radius-*, --blur-* stay as :root literals */
}
```

### 3.2 Contrast targets (WCAG AA — `08-design-system.md §8`, R-M4-06)

The light palette must clear these against **glass-over-gradient** (the actual rendered
surface, not a flat background):

| Surface | Token | Target | Note |
|---|---|---|---|
| Display numeral | `--text-on-scrim` on `--display-scrim` | ≥ 4.5:1 | Near-white scrim concentrates under the number — verify at gradient's darkest patch under the scrim |
| Digit button labels | `--text-primary` on `--glass-fill` over `--bg-aurora` | ≥ 4.5:1 | Glass alpha ~0.52 dilutes gradient; measure on-screen |
| Large operator labels | `--text-on-accent` on `--operator-accent` | ≥ 3:1 | White on coral-rose — the darkened `#e8285a` vs dark's `#ff5e7a` restores headroom |
| Equals button label | `--text-on-accent` on `--accent` | ≥ 3:1 | White on `#5b3de8` — verify this passes; darken further if needed |
| Focus ring | `var(--accent)` ring on light glass | Visible | Indigo ring on white glass — check against light glass background specifically |

BUILD must verify with a contrast checker on actual screenshots of both themes (not flat
color pickers — the gradient bleeds through the glass). If a target fails, **raise the
scrim/glass alpha** before modifying text color — readability wins over maximum
transparency (§8).
